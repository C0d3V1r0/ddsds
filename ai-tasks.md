### Задача 2.2 — Поиск инъекций и уязвимостей в коде

**Действие:** Проверь весь проект на:
- SQL-инъекции (конкатенация строк в запросах вместо параметризации).
- Command injection (вызов `os.system`, `exec`, `subprocess` с пользовательским вводом без санитизации).
- XSS (вставка пользовательских данных в DOM без экранирования, `dangerouslySetInnerHTML`).
- Path traversal (пользовательский ввод в путях к файлам без валидации).
- Небезопасная десериализация (`pickle.loads`, `eval`, `JSON.parse` без валидации схемы).
- Открытые эндпоинты без авторизации (API-роуты без middleware проверки токена/сессии).

**Результат:** Список уязвимостей:

Приоритет: критическая / средняя / низкая.

### Задача 4.6 — Декомпозиция всего кода

**Контекст:** Рефакторинг отдельных компонентов завершён (задачи 4.2–4.4). Теперь нужно пройтись по ВСЕМУ проекту целиком и убедиться, что каждый файл, каждая функция, каждый модуль — атомарны и имеют одну чёткую ответственность.

**Действие:** Пройди весь проект файл за файлом (Go, TypeScript, Python) и для каждого проверь:

1. **Файлы.** Каждый файл отвечает за одну сущность / один слой / одну задачу. Если файл содержит больше 200–300 строк — это сигнал к разбиению. Файл `utils.go` на 500 строк с 15 функциями — разбей на `retry.go`, `http.go`, `parse.go` и т.д. по смыслу.
2. **Функции и методы.** Каждая функция делает одно действие. Если функция длиннее 40–50 строк — разбей. Если у функции больше 3–4 параметров — вынеси параметры в структуру/интерфейс. Если функция содержит комментарии-разделители вроде `// --- шаг 2: валидация ---` — это готовые точки разбиения.
3. **Компоненты React.** Каждый компонент — одна визуальная/логическая единица. Если компонент содержит и логику получения данных, и бизнес-логику, и рендер — вынеси логику в хуки, данные в отдельный слой.
4. **Модули Python.** Каждый модуль — один домен. Если в одном файле и роутинг, и бизнес-логика, и работа с БД — разнеси по слоям (`routes/`, `services/`, `repositories/` или аналог, соответствующий `paradigms.md`).
5. **Пакеты Go.** Каждый пакет — одна зона ответственности. Если пакет `internal/` — свалка из 10 несвязанных файлов — реструктурируй.
6. **Общие типы и контракты.** Если одни и те же структуры данных / типы / интерфейсы дублируются между компонентами — вынеси в shared-слой или убедись, что они согласованы.

**Критерий проверки:** Открой любой файл в проекте — по его имени и первым 10 строкам должно быть понятно, что он делает. Если непонятно — файл нужно переименовать или разбить.

- Все тесты проходят. Все импорты обновлены.

### Задача 4.7 — Полная зачистка следов ИИ-генерации

**Контекст:** Проект мог быть частично или полностью написан с помощью ИИ. Цель — привести код в состояние, неотличимое от написанного живым профессиональным разработчиком. Никаких признаков генерации.

**Действие:** Пройди весь проект и найди/исправь характерные признаки ИИ-кода:

**1. Комментарии-паразиты (самый частый признак):**
- Удали комментарии, которые пересказывают код: `// создаём новый HTTP-клиент` перед `client := &http.Client{}`.
- Удали комментарии-заголовки в стиле «секций учебника»: `// ========== CONFIGURATION ==========`, `// --- Helper Functions ---`.
- Удали комментарии вида `// TODO: add error handling here` если рядом нет реального плана — это заглушки ИИ.
- Удали пояснения очевидных вещей: `// возвращаем результат` перед `return result`.
- Оставь только комментарии, которые объясняют ПОЧЕМУ, а не ЧТО: «почему таймаут именно 30 секунд», «почему тут ретрай 5 раз», «почему используем этот алгоритм, а не другой».

**2. Шаблонная структура кода:**
- Найди файлы, где все функции написаны по одному шаблону (одинаковая структура, одинаковые паттерны обработки ошибок copy-paste). Живой разработчик пишет неоднородно — где-то короче, где-то длиннее, в зависимости от контекста. Объедини шаблонный код в переиспользуемые хелперы.
- Найди избыточные обёртки, которые ничего не делают кроме проксирования вызова — удали.
- Найди чрезмерно «правильный» код: интерфейс с одной реализацией, абстрактная фабрика для создания одного объекта, паттерн ради паттерна — упрости.

**3. Нейминг:**
- Замени generic-имена, характерные для ИИ: `handleData`, `processItem`, `doWork`, `performAction`, `executeTask`, `utils`, `helpers`, `misc` — на конкретные имена, описывающие бизнес-домен.
