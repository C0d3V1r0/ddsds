# Парадигмы разработки в Nullius

## Коротко

Для Nullius принимаем подход **functional-first с точечным stateful OOP**.

Это значит:

- по умолчанию пишем код через **чистые функции**
- классы используем только там, где действительно есть **состояние, lifecycle или внешний runtime-адаптер**
- не строим архитектуру вокруг классов ради самих классов

## Когда использовать ФП

Функциональный стиль обязателен для кода, который:

- преобразует данные
- считает score и confidence
- готовит датасеты
- фильтрует, обогащает и валидирует события
- строит explainable detections
- работает как pipeline без внутреннего состояния
- легко тестируется как input -> output

В Nullius это прежде всего:

- `server/ml/features.py`
- `server/ml/baseline.py`
- `server/api/*`, где идёт формирование ответов
- risk scoring
- explainable enrichment
- форматирование и UI helpers во frontend

Почему:

- меньше скрытого состояния
- проще тестировать
- проще читать поток данных
- меньше лишней абстракции

## Когда использовать ООП

Лёгкое ООП допустимо и полезно только там, где есть:

- долгоживущее состояние
- lifecycle
- подключение к внешним системам
- обёртка над моделью или клиентом
- накопление runtime-контекста между вызовами

В Nullius это нормально для:

- `AnomalyDetector`
- `AttackClassifier`
- `Detector`, потому что он хранит окно SSH-попыток
- executor/client компонентов в агенте
- pydantic config models

Почему:

- там реально есть состояние
- функции без контейнера сделали бы код менее прозрачным

## Что считаем лишним ООП

В проекте не приветствуется:

- классы без состояния
- классы-обёртки вокруг 1–2 функций
- фабрики, наследование и абстракции без реальной пользы
- service/repository/manager слои “как в enterprise-шаблоне”, если они не упрощают код

Если модуль можно выразить как набор функций и структуры данных, делаем именно так.

## Практическое правило

Перед созданием класса задаём вопрос:

`Есть ли здесь состояние или lifecycle, которые действительно нужно инкапсулировать?`

Если ответ **нет**, пишем через функции.

Если ответ **да**, допускается небольшой класс без избыточной архитектурной тяжести.

## Текущий стандарт для проекта

- **Python backend**: functional-first
- **Go agent**: прагматичный stateful-стиль для runtime-компонентов, без лишней объектщины
- **React frontend**: функциональные компоненты, hooks и утилитарные функции

## Вывод

Nullius — не OOP-центричный проект.

Базовая парадигма проекта:

**ФП для логики и pipeline, ООП только для stateful runtime-компонентов.**
