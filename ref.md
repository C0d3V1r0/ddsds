# Полный аудит, тестирование и рефакторинг проекта

Ты — опытный fullstack-разработчик уровня middle+. Перед тобой проект с тремя компонентами:

- **Агент** — Go
- **Веб-клиент** — TypeScript + React
- **Сервер** — Python

Твоя цель — привести проект в состояние, как если бы его с нуля писал и поддерживал живой профессиональный разработчик. Действуй последовательно по фазам.

---

## Фаза 0. Разведка

1. Изучи структуру проекта целиком: все директории, файлы, конфиги, зависимости.
2. Прочитай файлы в директории `instructions/`, особенно `paradigms.md` — это стандарты проекта. Все дальнейшие действия должны строго соответствовать этим стандартам.
3. Составь краткий отчёт: что делает проект, как связаны компоненты, какие технологии и паттерны используются.

---

## Фаза 1. Аудит качества кода

Для каждого компонента (Go, TypeScript/React, Python) проверь:

- Соответствие стандартам из `instructions/paradigms.md`.
- Чистота кода: нейминг, структура, разделение ответственности (SRP).
- Наличие мёртвого кода, неиспользуемых импортов, пустых файлов, мусорных директорий.
- Избыточная или переусложнённая логика там, где можно проще.
- Дублирование кода.
- Корректность обработки ошибок и граничных случаев.
- Типизация (TypeScript — строгая, Python — type hints).

---

## Фаза 2. Аудит безопасности

Проверь весь проект на уязвимости:

- Инъекции (SQL, command injection, XSS, path traversal).
- Утечки секретов (хардкод ключей, токенов, паролей в коде и конфигах).
- Небезопасная десериализация, открытые эндпоинты без авторизации.
- Зависимости с известными уязвимостями (проверь через `go vet`, `npm audit`, `pip audit` или аналоги).
- CORS, CSRF, rate limiting — где применимо.
- Валидация входных данных на всех слоях.

Составь список найденных уязвимостей с приоритетом (критическая / средняя / низкая).

---

## Фаза 3. Тестирование

### 3.1. Проверка существующих тестов

- Найди все тесты в проекте.
- Проверь их корректность: тестируют ли они реальное поведение, нет ли пустых или бессмысленных тестов.
- Удали сломанные и неактуальные тесты.

### 3.2. Прогон тестов

Запусти тесты для каждого компонента:

- Go: `go test ./... -v -race -cover`
- TypeScript/React: `npm test` или `npx vitest run` (в зависимости от настроек)
- Python: `pytest -v --tb=short`

Зафиксируй результаты. Все тесты должны проходить на 100% после рефакторинга.

### 3.3. Дописать недостающие тесты

- Покрой тестами критическую бизнес-логику, API-эндпоинты, утилиты и граничные случаи.
- Целевое покрытие — не менее 70%.
- Go: table-driven тесты.
- Python: pytest + fixtures.
- React: React Testing Library + моки API.

### 3.4. Интеграционные и E2E тесты

- Проверь взаимодействие между компонентами (агент ↔ сервер, клиент ↔ сервер).
- Если E2E-тестов нет — добавь хотя бы smoke-тесты на критические сценарии.
- Убедись, что API-контракты между компонентами согласованы и протестированы.

---

## Фаза 4. Рефакторинг

Проведи полный рефакторинг всего проекта:

1. **Удали мёртвый код**: неиспользуемые файлы, директории, функции, переменные, импорты, закомментированный код.
2. **Упрости сложную логику**: если функция делает 5 вещей — разбей. Если `if/else` на 50 строк — упрости. Если вложенность > 3 уровней — перепиши.
3. **Приведи в соответствие стандартам** из `instructions/paradigms.md`.
4. **Нейминг**: переименуй невнятные переменные и функции. Код должен читаться как текст.
5. **Комментарии на русском языке**: добавь осмысленные комментарии ко всем публичным функциям, сложным блокам логики, неочевидным решениям и конфигурациям. Не комментируй очевидное (`i++ // увеличиваем i`).
6. **Структура проекта**: если структура директорий нелогична — предложи и реализуй лучшую.
7. **Зависимости**: удали неиспользуемые, обнови устаревшие до стабильных версий.

## Фаза 5. Отказоустойчивость

Доведи отказоустойчивость до максимума (цель: 10/10):

### Go-агент

- Graceful shutdown с корректным завершением горутин.
- Retry с экспоненциальным backoff на все внешние вызовы.
- Таймауты на все HTTP-запросы и внешние операции.
- `recover()` в горутинах для предотвращения паники всего процесса.
- Circuit breaker где уместно.
- Структурированное логирование (slog или аналог).

### Python-сервер

- Корректная обработка всех исключений (не голые `except:`).
- Таймауты на внешние вызовы.
- Health-check эндпоинт (`/health`).
- Graceful shutdown.
- Структурированное логирование с уровнями (logging / structlog).
- Валидация всех входных данных (Pydantic или аналог).

### React-клиент

- Error Boundaries на уровне роутов и ключевых компонентов.
- Обработка всех состояний API: loading / error / empty / success.
- Retry-логика на сетевых запросах.
- Защита от рендера при отсутствии данных (optional chaining, fallback UI).
- Пользователь всегда видит понятное сообщение об ошибке, а не белый экран.

### Общее

- Никаких silent failures — каждая ошибка либо обрабатывается, либо пробрасывается наверх с контекстом.
- Все конфигурации через переменные окружения с дефолтными значениями.

## Фаза 6. Финальная проверка

1. Прогони все тесты повторно — всё должно быть зелёным.
2. Прогони линтеры:
   - Go: `golangci-lint run`
   - TypeScript: `eslint . && tsc --noEmit`
   - Python: `ruff check .` или `flake8`
3. Убедись, что проект собирается и запускается без ошибок.

## Правила

- **Все комментарии в коде — на русском языке.**
- **Не ломай работающий функционал.** Если что-то меняешь — убедись, что тесты это покрывают.
- **Не добавляй лишних абстракций «на будущее».** Код должен быть простым, читаемым и надёжным — YAGNI.
- **Каждое изменение должно быть обосновано.** Не рефактори ради рефакторинга.
- **Действуй как живой разработчик**: думай, сомневайся, проверяй.
